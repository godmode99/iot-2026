Auth & Role System

ทุกคนใช้ Supabase Auth ชุดเดียวกัน ต่างกันที่ "type" ของ user และ "สิทธิ์" ในแต่ละ farm — ออกแบบมาสำหรับ production ตั้งแต่ต้น แต่ rollout ทีละ phase

User Types

🛡️

Super Admin

Phase 1
  • Pon + ทีมงาน (2–3 คน)
  • เห็นและทำได้ทุกอย่างในระบบ
  • bypass RLS ผ่าน service key
  • assign โดย Pon ใน Supabase dashboard
  • ไม่มี UI จัดการ — manual เท่านั้น
🌾

Customer (Farm Owner)

Phase 1
  • เจ้าของ farm ของตัวเอง
  • เห็นเฉพาะ device ใน farm ตัวเอง
  • จัดการ alert, ส่ง command ได้ทุกอย่าง
  • invite ลูกน้องเข้า farm ได้ (Phase 2)
  • RLS กั้น — ห้ามเห็น farm คนอื่น
👥

Farm Member

Phase 2
  • ถูก invite เข้า farm โดย owner
  • สิทธิ์กำหนดโดย owner ต่อคน
  • ดู / รับ alert / จัดการ alert / ส่ง command
  • ทำได้แค่สิ่งที่ owner อนุญาต
🤝

Reseller

Phase 3
  • ดูแล farm ของลูกค้าในมือตัวเอง
  • เห็นได้เฉพาะ farm ที่ถูก assign ไว้
  • read-only + support เท่านั้น
  • assign โดย super admin

สิ่งที่ตัดสินใจแล้ว (Lock แล้ว)

7 decisions นี้ปิดแล้ว — ห้ามเปลี่ยนโดยไม่อัปเดต Decision Register ก่อน ทุกคนในทีมต้องรู้และ follow

D-01ขนาด EnclosureClosed
ตัดสินใจ: ใช้ 150 × 100 × 60 mm เป็น Standard baseline สำหรับ PCB v1 และ pilot batch
เหตุผล: ล็อก mechanical baseline ไว้ก่อน ไม่ให้ PCB, antenna, และ waterproofing ล่อยลอย — ปรับ optimize หลัง pilot
ข้อห้าม: Long-Life ขยายได้แค่ battery bay ห้ามย้าย core board mount
D-02GPS FallbackClosed
ตัดสินใจ: ใช้ L76K เป็น default — สลับไป NEO-M8N เมื่อ TTFF > 60 วินาที หรือ error > 15m บ่อยเกินไป
เหตุผล: ประหยัดต้นทุนและพลังงาน รอให้ field data บอกก่อนว่าต้องอัปเกรดจริงไหม
D-03Provisioning FlowClosed
ตัดสินใจ: QR + Web/PWA เท่านั้น — ห้ามทำ BLE provisioning หรือ native app สำหรับลูกค้า
เหตุผล: flow เดียวทำให้ support, docs, และ firmware เรียบง่าย ทำงานได้ทุก mobile browser
D-044G ModemClosed
ตัดสินใจ: ใช้ A7670E เป็น default — SIM7670E เป็น fallback เฉพาะเมื่อ A7670E ขาดตลาดหรือแพงเกิน 20%
เหตุผล: รักษา flexibility ในการ source โดยไม่ปล่อยให้เปลี่ยน modem ตอนใกล้ launch โดยไม่ validate ก่อน
D-05LINE Notification PlanClosed
ตัดสินใจ: ใช้ LINE free tier ตลอด pilot — review เมื่อ forecast > 200 ข้อความ/เดือน หรือก่อน commercial launch
เหตุผล: ไม่จ่าย recurring cost ก่อนที่จะรู้ว่า pilot ส่ง notification มากแค่ไหน
D-06Battery PlatformClosed
ตัดสินใจ: core module เดียว + 2 battery variants (Standard และ Long-Life) — firmware/backend/provisioning ใช้ร่วมกัน
เหตุผล: ไม่แยก product line ทำให้ maintain ง่ายกว่า และ Long-Life เป็น optional upgrade ที่ไม่ทำให้ระบบหลักพัง
ข้อห้าม: Long-Life เป็น service-upgradeable เท่านั้น ลูกค้าเปิดกล่องเองไม่ได้
D-07Auth & Role ModelClosed
ตัดสินใจ: 4 user types (super_admin, reseller, customer, farm member) บน Supabase Auth ชุดเดียว — granular permission ต่อ farm member
เหตุผล: ออกแบบ production-grade ตั้งแต่ต้นเพื่อไม่ต้อง refactor ก่อน launch — rollout ทีละ phase ทำให้ pilot ไม่ช้า
Schema ใหม่: user_profiles, farm_members (Phase 2), reseller_farms (Phase 3)

Non-Negotiables — ห้ามเปลี่ยนโดยไม่ได้คุยกันก่อน

  • ลูกค้าทุกคนใช้ QR + Web/PWA เท่านั้น — ห้ามทำ BLE หรือ native app
  • ห้ามแยก firmware เป็นคนละ codebase ตาม battery variant
  • Long-Life เป็น service-upgradeable เท่านั้น — ลูกค้าเปิดกล่องเองไม่ได้
  • Standard ยังเป็น baseline หลักของ pilot ห้าม commit Long-Life ก่อนที่ enclosure และ battery pack นิ่ง
  • Operator ต้องสามารถดูและจัดการ device ของลูกค้าได้ทุก farm — ต้องมี super_admin access ก่อน pilot
