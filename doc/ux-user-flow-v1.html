  • trigger: sensor error flag หรือค่า invalid จาก ingest
  • auto-resolve: valid reading กลับมา

warning

📣 Notification Channels

  • LINE Free Tier: ส่งระหว่าง pilot — message พร้อม severity + device name
  • Email: structured stub รอ wire จริงก่อน launch
  • Web/in-app: alert badge + list บน Dashboard และ Ops page
  • rate-limit: ALERT_NOTIFY_MIN_INTERVAL_SEC กัน flood

🎛️ Alert Actions (UI)

  • Acknowledge: บันทึกว่าเห็นแล้ว alert ยัง open
  • Suppress: หยุด notify ชั่วคราว (ยัง open)
  • Resolve: ปิด alert manually + บันทึก resolved_at
  • ทุก action มี audit trail ใน system
  • UX: ปุ่มอยู่บน Device Detail Panel + Ops page
writable audit trail
⚠️ สิ่งที่ยังขาด / ต้องตัดสินใจ
  • Threshold config ยังอยู่ใน code — ถ้าอยากให้ operator ปรับได้ต้องมี UI เพิ่ม
  • Offline alert timing ควรต่างกันตาม battery-saving interval หรือเปล่า?
  • LINE notification ยังเป็น stub — ต้อง wire จริงก่อน pilot
  • Notification preference per-farm ยังไม่มี (EX-10 ระบุว่า out of scope MVP)

security
⚠️ สิ่งที่ยังต้องตัดสินใจ

⚙️ Commands Panel (OTA)

  • optional note สำหรับ audit

  • command log อัปเดตเมื่อ device ack: queued → sent → succeeded/failed
writable audit trail

📋 OTA Manifest Logic

GET /api/ota/manifest
  • backend ตรวจ: current_version, channel, battery_variant
  • filter releases ตาม channel และ battery_variants
  • เลือก release ที่ version สูงกว่า current เท่านั้น
  • ถ้าไม่มี release ใหม่ → ไม่ต้อง update
read-only system logic

📊 Command Log States

  • queued: operator สั่งแล้ว รอ device รับ
  • sent: backend ส่ง command ไปยัง device แล้ว
  • succeeded: device ack สำเร็จ
  • failed: device report error หรือ timeout
  • cancelled: operator ยกเลิกก่อน device รับ
queued sent succeeded failed

⚠️ OTA Safety Rules (UX)

  • ota_apply ควรมี confirm dialog ก่อน — device จะ reboot
  • แสดง firmware version ปัจจุบัน vs version ใหม่ก่อน apply
  • ระหว่าง OTA device จะ offline ชั่วคราว — dashboard ไม่ควร trigger offline alert ทันที
  • ถ้า OTA ล้มเหลว firmware rollback อัตโนมัติ (device side)
  • log ทุก OTA action พร้อม requested_by user ID
safety confirm required
⚠️ สิ่งที่ยังต้องตัดสินใจ
  • OTA progress ปัจจุบัน track ผ่าน command_log status เท่านั้น — ถ้าอยากให้ real-time ต้องมี polling หรือ websocket
  • Batch OTA (หลาย device พร้อมกัน) ยังเป็น SVC route ไม่มี UI
  • Offline alert suppression ระหว่าง OTA ยังไม่มี logic — ควรเพิ่ม grace period หลัง ota_apply
  • Release catalog ปัจจุบัน load จาก JSON file — production ควรเป็น DB หรือ blob storage
